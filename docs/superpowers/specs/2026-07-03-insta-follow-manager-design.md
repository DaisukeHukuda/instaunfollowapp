# Instagram フォロー整理アプリ 設計書

作成日: 2026-07-03
ステータス: ユーザー承認済み（会話でのレビュー完了）

## 1. 背景と目的

自分のInstagramアカウントのフォロー/フォロワー状況を可視化し、不要なアカウントを効率よくアンフォローできるようにする、**PCローカル専用**のWebアプリ。

### 重要な前提制約

- Instagramには「フォロー一覧の取得」「アンフォロー実行」ができる公式APIが**存在しない**。
- 本アプリは規約違反・アカウント凍結リスクを避けるため、以下の方針を採る:
  - データ取得は**公式エクスポート**（設定 → 「情報をダウンロード」→ **JSON形式**を選択）のZIPを使う。
  - アンフォロー/フォローの**実行は自動化しない**。アプリはプロフィールページを新しいタブで開き、本人がInstagram上でタップする（計2クリック）。アプリ側は「アンフォロー済み」の記録だけを持つ。
  - プロフィール情報（写真・自己紹介など）の補完のみ、本人のログインCookieを使った**低速・読み取り専用**の取得を行う（詳細は §6）。

## 2. スコープ

### やること

- エクスポートZIPの解析と関係性の自動分類（相互 / 片思い / ファン）
- 写真・自己紹介つきの見やすいアカウント一覧（絞り込み・並び替え・検索）
- 各アカウントの「プロフィールを開く」「アンフォロー済み」「残す」操作
- 複数選択 → レビューキューでテンポよく一括整理（キーボードショートカット対応）
- プロフィール自動取得（進捗バー・自動一時停止・フォールバック付き）
- 2回目以降のZIP取り込みでの差分表示（実際に減ったフォロー、新規フォロワー等）

### やらないこと（非スコープ）

- アンフォロー/フォローの自動実行（規約違反のため恒久的に対象外）
- スマホ対応・外部公開デプロイ（将来の拡張候補。今回は対象外）
- 複数Instagramアカウントの管理
- DM・投稿・ストーリー等、フォロー関係以外のデータ

## 3. アーキテクチャ

案B「ローカルサーバ一体型」を採用。

```
[ブラウザ] React SPA (Vite build)
    │ fetch /api/...
[ローカルサーバ] Node + Hono  ← npm start で起動、localhost:3900
    ├─ 静的ファイル配信（SPA）
    ├─ ZIP解析・関係性分類
    ├─ プロフィール取得ワーカー（低速・直列）
    └─ 永続化: data/ フォルダ（JSONファイル + 画像キャッシュ）
```

- 技術: TypeScript / Node 20+ / Hono / Vite + React
- テスト: Vitest（パーサ・分類・差分ロジック）
- UI文言: 日本語
- データはすべて `data/` 配下に保存。`data/` は `.gitignore` に含める（フォロワー情報・Cookieを絶対にコミットしない）。

## 4. データモデル

`data/accounts.json` — アカウントの一覧（username がキー）:

| フィールド | 型 | 説明 |
|---|---|---|
| username | string | Instagramユーザー名（一意キー） |
| profileUrl | string | `https://www.instagram.com/<username>/` |
| relationship | `mutual` \| `followingOnly` \| `followerOnly` | 相互 / 片思い / ファン |
| followedAt | string(ISO) \| null | 自分がフォローした日時（エクスポート由来） |
| followerSince | string(ISO) \| null | 相手にフォローされた日時 |
| status | `pending` \| `unfollowed` \| `followedBack` \| `keep` | 未処理 / アンフォロー済み(自己申告) / フォローした(自己申告・ファン向け) / 残す |
| statusChangedAt | string(ISO) \| null | 状態変更日時 |
| queued | boolean | レビューキューに入っているか |
| profile | object \| null | 補完データ（下記） |

`profile` オブジェクト: `displayName`, `bio`, `followerCount`, `followingCount`, `postCount`, `isPrivate`, `isVerified`, `picPath`（ローカル画像パス）, `fetchedAt`, `fetchError`（失敗時の理由）。

### 関係性分類ロジック

- 相互（mutual）= フォロー中 ∩ フォロワー
- 片思い（followingOnly）= フォロー中 − フォロワー ← **アンフォロー候補の筆頭**
- ファン（followerOnly）= フォロワー − フォロー中 ← フォローバック候補

### 再取り込み時の突き合わせ（差分）

2回目以降のZIP取り込みで:

1. フォロー中リストから消えたアカウント → status を `unfollowed`（確定）に更新
2. `unfollowed` にしたのにまだフォロー中リストにいる → `pending` に戻し「未完了」バッジ表示
3. `followedBack` にしたアカウントがフォロー中リストに現れた → relationship を `mutual` に更新（確定）
4. 新規フォロワー / 新規フォロー / 離脱フォロワーを差分サマリーとして表示
5. `status: keep` と `profile` は取り込みをまたいで維持する

## 5. API（サーバ）

| メソッド/パス | 役割 |
|---|---|
| `POST /api/import` | ZIPアップロード → 解析 → accounts.json 更新 → 差分サマリー返却 |
| `GET /api/accounts` | 一覧取得（relationship / status / queued / 検索語 / ソートのクエリ対応） |
| `PATCH /api/accounts/:username` | status・queued の更新 |
| `POST /api/queue/bulk` | 複数アカウントの一括キュー投入/解除 |
| `POST /api/enrich/start` `POST /api/enrich/stop` `GET /api/enrich/status` | プロフィール取得の開始/停止/進捗（SSEまたはポーリング） |
| `POST /api/settings/cookie` | Cookie登録（検証してから保存） |
| `GET /api/stats` | 件数サマリー・最新取り込み差分 |

## 6. プロフィール自動取得（enricher）の設計

**このアプリで唯一リスクに触れる箇所。安全側に倒す。**

- 取得元: `https://i.instagram.com/api/v1/users/web_profile_info/?username=<name>`（本人のセッションCookie + 標準ヘッダを付与した読み取り専用GET）
- ペース: **1件ずつ直列、間隔3〜5秒（ランダムジッター付き）**。並列取得は実装しない
- 画像: プロフィール画像はダウンロードして `data/profiles/` に保存（InstagramのCDN URLは期限切れになるため、URL保存ではなくファイル保存が必須）
- キャッシュ: 取得済み（`fetchedAt` あり）はスキップ。再取得は明示操作でのみ
- 停止条件:
  - HTTP 401/403（Cookie失効）→ 即時停止し「Cookieを更新してください」を表示
  - HTTP 429 または連続3件失敗 → 自動一時停止（クールダウン後に手動再開）
- フォールバック: 取得失敗したアカウントはユーザー名＋関係性バッジのみで表示し、アプリの他機能は一切影響を受けない
- Cookie保存: `data/secrets/cookie.json`（gitignore対象）。UIには値を再表示しない

## 7. 画面設計（3画面）

### ① 一覧画面（メイン）

- アカウントカードのグリッド表示: 写真（なければイニシャル）・ユーザー名・表示名・自己紹介(先頭2行)・関係性バッジ・フォロー日・フォロワー数
- フィルタ: 関係性（相互/片思い/ファン）、status（未処理/済み/残す）、検索（ユーザー名・表示名・bio）
- ソート: フォロー日の古い順/新しい順、フォロワー数
- カード操作: プロフィールを開く（新規タブ）/ アンフォロー済み / 残す。ファン（followerOnly）には「フォローした」ボタンも表示
- チェックボックス複数選択 → 「選択をキューに入れる」

### ② レビューキュー画面

- キュー内アカウントを1件ずつ大型カード表示、進捗（例: 12/45）
- キーボードショートカット: `O`=プロフィールを開く / `U`=アンフォロー済みにして次へ / `K`=残すにして次へ / `→`=スキップ
- 中断・再開可能（queued フラグと status が永続化されているため）

### ③ 取り込み・設定画面

- ZIPドラッグ&ドロップ + 解析結果サマリー（件数・差分）
- エクスポートの取得手順ガイド（**JSON形式を選ぶこと**を明記）
- プロフィール取得: Cookie登録フォーム、開始/停止ボタン、進捗バー、失敗一覧
- 統計: 相互/片思い/ファンの件数、前回取り込みとの差分

## 8. エラー処理

| ケース | 挙動 |
|---|---|
| ZIPに followers/following ファイルがない | 「HTML形式でエクスポートされた可能性。JSON形式で再申請を」と具体的に案内 |
| ZIP構造のバリエーション | エクスポートのファイル名揺れ（`followers_1.json` 分割等）に対応。パス直書きせず走査で発見 |
| Cookie失効（401/403） | enricher即時停止 + 画面通知。既存データは保持 |
| レート制限（429/連続失敗） | 自動一時停止。手動再開まで待機 |
| accounts.json 破損 | 起動時に検出し、直前の取り込みスナップショットから再構築を提案 |

## 10. テスト方針

- Vitest によるユニットテスト: ZIP/JSONパーサ、関係性分類、再取り込み差分ロジック（このアプリの信頼性の核）
- enricher はレスポンスをモックした単体テスト + 実データでの少数件手動確認
- UIは各ステップ末の手動確認（ローカル専用・単一ユーザーのためE2E自動化はしない）

## 11. 実装体制

- 実装作業は Sonnet/Opus のサブエージェントに切り出し、メインセッション（Fable 5）は設計・レビュー・監査に専念する
- 例外として、Cookie取り扱い・enricher のペース制御など安全性に直結する箇所はメインセッションが直接実装または精査する
- Git: ローカルで `git init` 済み。GitHubリモートはユーザーから後日提供され次第接続する

## 12. 決定事項の経緯（要約)

- 実現方式: 公式エクスポート + 手動アンフォロー（規約違反ゼロを優先）
- 表示: 関係性分類 + 公開プロフィール取得（Cookie利用の低速読み取り）
- 実行環境: PCローカルのみ（プライバシー優先）
- 一括UX: レビューキュー方式
- 構成: 案B ローカルサーバ一体型
